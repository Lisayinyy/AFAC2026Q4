## 一、赛题与规则要点（含论坛官方澄清）

- 5 领域 86 份文档（保险16 / 监管26 / 合同14 / 财报10 / 研报20），A/B 榜各 100 题；题型：多选65 / 判断20 / 单选15（A榜实测分布）
- 评分：`FinalScore = 100 × Accuracy × (0.7 + 0.3 × TokenScore)`，`TokenScore = (5M − TotalTokens)/5M`
- **多选题无部分分**，漏选/多选/错选全错 → 多选题是主战场（65%题量）
- 推理阶段只能用 Qwen 系列（百炼/魔搭 API），**禁止一切 embedding 模型**（含 qwen embedding，论坛官方确认"不能用向量"）
- ⭐ **官方论坛关键澄清（2026-06-09，官方人员郑宁）：数据预处理阶段的 token 消耗不计入总消耗量**
  - 含义：可以在预处理阶段用 Qwen 大方地做文档结构化、建索引，只要不生成"语义摘要/FAQ/知识库直接用于答题"（规则红线）
  - 来源：tianchi.aliyun.com/forum/issueDetail?postId=1050799
- Token 消耗为选手自报，论坛已有刷分争议（A榜有100分队伍）→ 我们按规则诚实统计，B榜前15要交代码审核，刷分队伍会被清退
- 每天提交 3 次
- 官方 baseline 很弱：A榜 49%（token 299万），B榜 13% → 提升空间大
- 基准模型 Qwen3.6-plus

## 二、外部方案调研结论

### 2.1 同类比赛冠军方案（可直接借鉴）

| 方案 | 核心做法 | 对我们的启示 |
|---|---|---|
| FinGLM 年报问答冠军（馒头科技） | 问题分类 + 关键词逐行匹配表格 + **计算题匹配公式模板→Python 外部计算**，LLM 只组织语言 | 财报数值题不让 LLM 心算 |
| FinGLM 第三名（零微调） | 规则问题分类 + 正则关键词检索（决赛放弃向量、**纯关键词也拿奖**）+ 同义词泛化 | 印证禁 embedding 下 BM25/关键词可行 |
| AFAC2024 金融智能体冠军 | difflib/LCS 多路字符串召回→LLM 精选；5 套差异化 prompt **多数投票**；temperature=0 | 实体归一、投票稳分 |
| Enterprise RAG Challenge 2025 冠军 | Docling 解析+表格转HTML；小块只当"指针"命中后**回溯整页**；1文档1索引+元数据路由；**LLM Reranker**（0-1打分，3页/批）；按答案类型路由4套prompt；比较题拆子问题 | 父页面检索 + LLM精排 + 题型路由是最接近本赛题的完整参考（github.com/IlyaRice/RAG-Challenge-2） |

### 2.2 Token 压缩技术

- **LongLLMLingua**：问题感知压缩，4x 压缩性能反升 21.4%（arxiv 2310.06839）——思想可用 Qwen 实现（用便宜模型删无关句）
- **RECOMP**：检索后先压缩再入 prompt，压到 6% 仍几乎无损（arxiv 2310.04408)
- **记忆流转模式**：scratchpad 笔记、渐进式披露（不灌全文，Agent 自主逐层下钻）、map-reduce over chunks
- **百炼降本开关**：Batch 调用半价；qwen3 系列上下文缓存折扣；新开通 90 天送额度

### 2.3 Qwen 模型选型（百炼 2026-07 价格，元/百万token 输入/输出）

| 模型 | 价格 | 用途 |
|---|---|---|
| qwen-flash | 0.15 / 1.5（≤128K档） | 证据段打分/粗筛（约为旗舰 1/80 成本） |
| qwen-turbo | 0.3 / 0.6 | 备选粗筛 |
| qwen-long | 0.5 / 2（10M 上下文，不阶梯涨价） | 整文档读取兜底 |
| qwen3.6-plus（评测基准） | ~0.8 / 2-8 阶梯 | 最终答题主力 |
| qwen3.7-max | 12 / 36 | 只用于疑难题仲裁，慎用 |

注意：plus/flash 超过 128K/256K 档后单价跳涨 3-6 倍 → **先压缩再进大档位**。

## 三、我们的技术方案（三级漏斗 + 分题型策略）

### 3.1 总体架构

```
预处理（token 不计费，可重度使用 Qwen）
  PDF/HTML 解析（pdfplumber，已完成 22,599 chunks）
  → 结构化索引：文档元数据表（公司/年份/法规号/产品名）+ 章节目录树 + 条款号倒排
  → 表格抽取为结构化数据（财报三表、保险费率表）

推理（计费，三级漏斗控 token）
  L1 文档路由（B榜）：元数据规则匹配 + BM25 → 候选文档 ≤4
  L2 段落检索：BM25+条款号/数字精确串加权 → 候选块 ~30
  L3 LLM 精排：qwen-flash 对候选块 0-1 打分（批量）→ 精选 ~8 块
  答题：qwen3.6-plus，temperature=0，结构化输出（reasoning + answer）
  后处理：字母抽取/排序/去重/合法性校验（已完成）
```

### 3.2 分题型策略

| 题型 | 策略 |
|---|---|
| 多选（65题，主战场） | **逐选项独立验证**：每个选项转成判断题单独配证据核验 → 汇总；temperature=0 + 3 次自洽投票，不一致的题加大证据量重跑 |
| 判断（20题） | 单次验证 + 反向验证（"找出与该陈述矛盾的原文"） |
| 单选（15题） | 标准 CoT + 证据引用 |
| 财报计算题 | 抽数（LLM 只负责从表格取数）→ **Python 计算** → 比对选项 |
| 跨文档比较题 | 拆子问题分别检索，各文档独立取证后合并判断 |

### 3.3 Token 预算分配（5M 总额）

- 目标：100 题 ≤ 3M（留 40% TokenScore ≈ +12 分加权）
- 单题预算：普通题 ~15k，多选验证题 ~30k，疑难题上限 50k
- 省钱手段：L3 精排用 qwen-flash；答题上下文只放精选块；批量接口半价；失败重试上限 2 次

### 3.4 里程碑（A 榜剩 13 天）

| 时间 | 目标 |
|---|---|
| D1-2（7/8-9） | 管线跑通首次提交（已完成框架+预处理，差 API key）；拿到 baseline 分 |
| D3-5 | 结构化索引 + LLM 精排 + 逐选项验证上线；目标 A 榜 65%+ |
| D6-9 | 财报计算链路 + 分领域 prompt 精调 + 错题分析迭代；目标 75%+ |
| D10-12 | 自洽投票 + token 瘦身（压缩/缓存/batch）；目标 80%+ 且 ≤3M token |
| D13（7/21前） | 冻结代码，B 榜文档路由压测（B榜不给 doc_ids，13%→路由是B榜生死线） |

### 3.5 风险与对策

- **B 榜断崖**（baseline 49%→13%）：文档路由是最大风险 → 预处理阶段给每份文档建"身份卡"（公司/产品/法规号/年份/关键词表），路由先规则后 LLM
- 多选无部分分：宁可保守（证据不足时选最高置信组合），逐选项验证降低连坐
- Token 自报刷分者扰乱排名：不跟风，B 榜审核会清退
- PDF 表格解析质量：财报题错误多半源于此，必要时 MinerU 重解析财报 10 份

## 四、待队友讨论的决策点

1. 答题主模型用 qwen3.6-plus（对齐官方基准）还是 qwen-long（便宜且长）？建议：precision 关键题用 plus，长上下文兜底用 long
2. 预处理阶段是否用 Qwen 给每个 chunk 打领域标签/抽关键词（token 免费但注意"不得生成语义摘要直接用于答题"的红线怎么界定——建议只做**索引性标注**不做内容摘要）
3. 每天 3 次提交怎么用：建议 1 次留给当日最优，2 次做 AB 实验
4. 分工：解析&索引 / 检索&路由 / 答题&prompt / 评测&错题分析
